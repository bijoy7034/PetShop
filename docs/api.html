Pet Shop Wholesale Inventory API

FastAPI + MongoDB backend. 31 endpoints across 8 domains. All state-changing endpoints require an auth cookie + CSRF header.

Stack & base URL

Base URLhttp://localhost:8000/api
AuthHttpOnly cookie petshop_access (JWT) + petshop_refresh + petshop_csrf
CSRF headerX-CSRF-Token (value = petshop_csrf cookie) — required on POST/PATCH/DELETE
Content typeapplication/json (except POST /products/bulk-upload: multipart/form-data)
Rolesadmin, office_staff, sales_rep

Auth flow

  1. Set ADMIN_EMAIL/ADMIN_PASSWORD/ADMIN_NAME in backend/.env — the first admin is seeded on startup with must_change_password: true.
  2. POST /api/auth/login with email+password. Server sets 3 cookies. Response body has the user profile + TTL.
  3. Admin calls POST /api/users to create office_staff and sales_rep accounts (server returns a one-time temp password).
  4. New users log in and immediately POST /api/auth/change-password.
  5. The frontend axios interceptor reads petshop_csrf cookie and sends it back as X-CSRF-Token on every mutation.
cURL note. The examples below assume you have logged in and captured cookies in cookies.txt. Every write flag pattern is: 
curl -b cookies.txt -c cookies.txt \
     -H "Content-Type: application/json" \
     -H "X-CSRF-Token: $(grep petshop_csrf cookies.txt | awk '{print $NF}')" \
     -X POST http://localhost:8000/api/… -d '…json…'

Sample: login

curl -c cookies.txt -H "Content-Type: application/json" \
  -X POST http://localhost:8000/api/auth/login \
  -d '{"email":"owner@petshop.local","password":"OwnerPass1234"}'

Response 200

{

Both are first-class collections. Products reference a subcategory; the server auto-fills the parent category_id on the product from the subcategory. Frontend only needs a "pick a subcategory" dropdown.

// (name, category_id) is unique — same name can exist under different categories.

POST /api/subcategories  { "name": "Wet food",   "category_id": "{dog_food_id}" }
POST /api/subcategories  { "name": "Dry kibble", "category_id": "{cat_food_id}" }
POST /api/subcategories  { "name": "Chew",       "category_id": "{toys_id}" }

3b) Or bulk-upload from Excel

Headers accepted (case-insensitive): name, category, subcategory, description, base_price, discount_price, variant_size, variant_weight, variant_color, variant_sku, variant_price, variant_discount_price, variant_stock.

Rows sharing the same name are merged into one product with multiple variants. Both category AND subcategory are required — the sheet resolves them by name (case-insensitive on subcategory) and rejects rows referencing unknown ones. Categories and subcategories must exist beforehand (create them via the API before uploading).

4) Sales rep adds a store

// Login as sales_rep first, then:
POST /api/stores
{
  "name": "Happy Pets Corner",
  "location": "MG Road, Bengaluru",
  "contact": {
    "name": "Rohit Menon",
    "phone": "+91-99450-33333",
    "email": "rohit@happypets.example"
  },
  "geo": { "lat": 12.9716, "lng": 77.5946 },
  "email": "orders@happypets.example",
  "gst_number": "29ABCDE1234F1Z5",
  "notes": "Prefers deliveries before 11am"
}

Store is created with status pending. Sales rep cannot place orders or mark attendance until office approves.

5) Office approves the store

// Login as office_staff, then:
POST /api/stores/{store_id}/approve
{ "credit_limit": 100000 }

6) Sales rep visits the store and marks attendance

POST /api/attendance
{
  "store_id": "{store_id}",
  "lat": 12.9717,      // within 100m of stored 12.9716
  "lng": 77.5947,
  "notes": "Weekly visit; discussed new arrivals"
}

Fails with 400 if outside 100m or if store status ≠ approved.

7) Sales rep places an order

POST /api/orders
{
  "store_id": "{store_id}",
  "notes": "Monthly restock",
  "lines": [
    { "product_id": "{pedigree_id}", "variant_id": "{ped_3k_id}",  "qty": 5 },
    { "product_id": "{whiskas_id}",  "variant_id": "{wsk_1200_id}", "qty": 8 },
    { "product_id": "{kong_id}",     "variant_id": "{kong_m_id}",  "qty": 4 }
  ]
}

Server computes total, checks credit_used + total ≤ credit_limit, holds the credit, and creates the order in placed status.

8) Office progresses the order

POST /api/orders/{order_id}/accept    // decrements variant stock atomically
POST /api/orders/{order_id}/pack
POST /api/orders/{order_id}/dispatch
POST /api/orders/{order_id}/deliver

Cancel (sales rep, only before accept)

POST /api/orders/{order_id}/cancel
{ "reason": "Store changed their mind" }

End-to-end test recipe

  1. Start Mongo & the backend. Confirm GET /api/health → 200.
  2. Log in as admin (from env-seeded account). Change password.
  3. Create office and sales-rep accounts. Log each in on a separate cookie jar and change passwords.
  4. As office: create categories + subcategories, then products with variants.
  5. As sales_rep: create a store. As office: approve with a credit limit.
  6. As sales_rep: mark attendance (test both success within 100m and failure outside).
  7. As sales_rep: place an order under the credit limit. Try one over — must fail.
  8. As sales_rep: cancel the order. Verify credit_used on the store dropped back to 0.
  9. Place a fresh order. As office: accept → pack → dispatch → deliver. After accept, confirm variant stock decremented.
  10. GET /api/orders/{id} — inspect history[] for the full audit trail.

Health

GET /api/health public

Ping the service. Also pings Mongo.

Response 200

{ "status": "ok", "MongoDB": "ok", "message": "Service is healthy" }

Auth

POST /api/auth/login public

Exchange email + password for a session. Sets three cookies.

Body

{ "email": "owner@petshop.local", "password": "OwnerPass1234" }

Response 200

{ "token_type": "cookie", "expires_in_seconds": 900, "user": { ... } }

Errors

  • 401 — bad email/password
  • 403 — account inactive

POST /api/auth/refresh public

Rotate the session using the refresh cookie. Returns a new access token and rotated refresh cookie. Reusing a rotated refresh wipes all sessions for that user.

GET /api/auth/me any user

Fetch the caller's profile.

POST /api/auth/change-password any user

Rotate the caller's password. Revokes all other sessions on success.

Body

{ "current_password": "old", "new_password": "at-least-ten" }

POST /api/auth/logout any user

Revoke the current session and clear cookies.

POST /api/auth/logout-all any user

Revoke every session for the caller (across all devices).

List users. Query: role, status, search, page, page_size.

GET /api/users/{user_id} any user

Create a new user. If password is omitted, server generates a one-time password and returns it; the account is flagged must_change_password.

Body

  "email": "office@petshop.local",
  "name": "Priya (Office)",
  "role": "office_staff",
  "phone": "+91-98450-11111"
}

Response 201

{
  "user": { "_id": "...", "email": "office@petshop.local", "role": "office_staff", ... },
  "temporary_password": "aB3xYz…"
}

PATCH /api/users/{user_id} admin

Partial update. Fields: name, role, phone, status, password.

DELETE /api/users/{user_id} admin

Hard-delete. Refuses if the caller is deleting themselves.

Subcategories

First-class collection. Each subcategory belongs to exactly one category. (name, category_id) is unique — the same name can exist under different categories.

GET /api/subcategories any user

Flat list — this is what the frontend product-create dropdown queries. Query: category_id (filter to one category), search, page, page_size. Response items include the denormalised category_name so you can render "Dog Food → Dry kibble" without a join.

Response 200 (excerpt)

{
  "items": [
    { "_id": "6a...", "name": "Dry kibble", "category_id": "...",
      "category_name": "Dog Food", "description": null,
      "created_at": "...", "updated_at": "..." }
  ],
  "total": 4, "page": 1, "page_size": 200
}

GET /api/subcategories/{sub_id} any user

POST /api/subcategories office

Body

{
  "name": "Dry kibble",
  "category_id": "{dog_food_id}",
  "description": "Optional"
}

Errors

  • 400 — unknown category_id
  • 409 — a subcategory with that name already exists under this category

PATCH /api/subcategories/{sub_id} office

Rename or move to a different category. A rename propagates to every product tagged with this subcategory (denormalised subcategory_name is updated). Moving to another category rewrites category_id and category_name on the affected products too.

DELETE /api/subcategories/{sub_id} office

Products previously tagged with this subcategory will retain their stored subcategory_id/subcategory_name — they'll effectively point at a deleted row. Consider re-tagging affected products first.

Attendance

POST /api/attendance sales_rep

Mark a visit. Server verifies the rep owns the store, the store is approved, and the caller is within 100 m of the store's saved geo.

Body

{
  "store_id": "{store_id}",
  "lat": 12.9717,
  "lng": 77.5947,
  "notes": "Weekly visit"
}

Errors

  • 404 — store not found or not owned by caller
  • 400 — store not approved
  • 400 — outside 100 m (message includes actual distance)

GET /api/attendance any user

Sales rep sees only their own records (the user_id query param is ignored for them). Office/admin can filter freely. Query: store_id, user_id, page, page_size.

Orders

GET /api/orders any user

Sales rep sees only their own orders. Office/admin see all. Query: store_id, status, page, page_size.

GET /api/orders/{order_id} owner sales_rep or office

Returns the full order incl. history[] — every status transition with actor + timestamp. This is what the office order detail UI should render as a step tracker.

POST /api/orders sales_rep

Place an order. Server prices every line, sums the total, and enforces credit_used + total ≤ credit_limit. Credit is held until cancel or write-off.

Body

{
  "store_id": "{store_id}",
  "notes": "Monthly restock",
  "lines": [
    { "product_id": "{pedigree_id}", "variant_id": "{ped_3k_id}",   "qty": 5 },
    { "product_id": "{whiskas_id}",  "variant_id": "{wsk_1200_id}", "qty": 8 }
  ]
}

{
  "_id": "6a12…",
  "store_id": "…",
  "owner_id": "…",
  "status": "placed",
  "total": 9710.00,
  "lines": [

Errors

  • 404 — store not owned by caller
  • 400 — store not approved / variant not found / insufficient stock / credit exceeded

POST /api/orders/{order_id}/cancel owner sales_rep or office

Only allowed while status is placed. Releases the credit hold.

Body

{ "reason": "Store changed their mind" }

POST /api/orders/{order_id}/accept office

Transitions placed → accepted and atomically decrements variant stock for every line. If any line underflows mid-way (concurrent order), rolls back and returns 409.

POST /api/orders/{order_id}/pack office

accepted → packing

POST /api/orders/{order_id}/dispatch office

packing → out_for_delivery

POST /api/orders/{order_id}/deliver office

out_for_delivery → delivered
