Pet Shop Wholesale Inventory API

FastAPI + MongoDB backend. All state-changing endpoints require an auth cookie + CSRF header.

Recent changes (2026-07 batch).
  • Product and Variant both gained is_active. Toggle via dedicated PATCH .../toggle-active endpoints — regular PATCH doesn't touch it. Order placement rejects inactive items.
  • Variant keeps a price_history[]. PATCH pushes the previous (price, discount_price) when either changes, with the optional reason.
  • OrderLine.placed_qty → qty_ordered. accepted_qty → qty_accepted. Line also stores discount_price so historical pricing survives future variant changes.
  • New order statuses: pending_admin_approval and delayed. Over-credit orders no longer reject — they land in pending_admin_approval with inventory reserved but credit NOT held. Admin approves or rejects.
  • Order snapshots at placement: store_district, delivery_address_snapshot, credit terms (credit_period_days), cancellation policy (is_free_cancellation, cancellation_charges), return_window_days. Historical orders survive future store edits.
  • Store credit-limit changes now require admin approval: propose via PATCH .../credit-limit, admin approves or rejects.
  • Attendance → Visits. Multiple visits/day per store now allowed. New fields: mode (in_store/remote), outcome, order_value, duration_minutes.
  • Inventory keeps stock_history[] with previous/new/delta/reason/actor/timestamp on every on-hand change.

Auth flow

  1. Set ADMIN_EMAIL/ADMIN_PASSWORD/ADMIN_NAME in backend/.env — the first admin is seeded on startup with must_change_password: true.
  2. POST /api/auth/login with email+password. Server sets 3 cookies. Response body has the user profile + TTL.
  3. Admin calls POST /api/users to create office_staff and sales_rep accounts (server returns a one-time temp password).
  4. New users log in and immediately POST /api/auth/change-password.
  5. The frontend axios interceptor reads petshop_csrf cookie and sends it back as X-CSRF-Token on every mutation.
cURL note. The examples below assume you have logged in and captured cookies in cookies.txt. Every write flag pattern is: 
curl -b cookies.txt -c cookies.txt \
     -H "Content-Type: application/json" \
     -H "X-CSRF-Token: $(grep petshop_csrf cookies.txt | awk '{print $NF}')" \
     -X POST http://localhost:8000/api/… -d '…json…'

Sample: login

curl -c cookies.txt -H "Content-Type: application/json" \
  -X POST http://localhost:8000/api/auth/login \
  -d '{"email":"owner@petshop.local","password":"OwnerPass1234"}'

Both are first-class collections. Products reference a subcategory; the server auto-fills the parent category_id on the product from the subcategory. Frontend only needs a "pick a subcategory" dropdown.

// Subcategories are separate documents that reference a category_id.
// (name, category_id) is unique — same name can exist under different categories.

POST /api/subcategories  { "name": "Chew",       "category_id": "{toys_id}" }

3) Products with auto-generated variants (office)

You define the product's option set (arrays of size / weight / color values). The server expands the Cartesian product of every non-empty axis into individual variants, each seeded at price = base_price, stock = 0, sku = null. Office then fills real prices and SKUs per variant via PATCH, and initial stock via the adjust-stock endpoint.

// Single-axis: 3 sizes -> 3 variants auto-generated
POST /api/products
{
  "name": "Pedigree Adult Chicken & Vegetables",
  "subcategory_id": "{dog_dry_kibble_id}",
  "description": "Adult dog food with real chicken and vegetables.",
  "base_price": 350,
  "option_sets": { "size": ["1kg", "3kg", "10kg"] }
}

// Two axes: 3 sizes x 2 colors -> 6 variants
POST /api/products
{
  "name": "Kong Classic Rubber Toy",
  "subcategory_id": "{toy_chew_id}",
  "base_price": 550,
  "option_sets": { "size": ["Small", "Medium", "Large"], "color": ["Red", "Blue"] }
}

// No option_sets -> single default variant with all axes null (single-SKU product)
POST /api/products
{
  "name": "Petkin Dog Wipes",
  "subcategory_id": "{grooming_wipes_id}",
  "base_price": 220
}

Then fill each variant's real price / SKU (loop over the returned variants):

PATCH /api/products/{product_id}/variants/{variant_id}
{ "sku": "PED-CHK-1K", "price": 350, "discount_price": null }

POST  /api/products/{product_id}/variants/{variant_id}/adjust-stock
{ "delta": 200, "reason": "Initial receipt from supplier" }

7) Sales rep places an order

POST /api/orders
{
  "store_id": "{store_id}",
  "notes": "Monthly restock",
  "lines": [
    { "product_id": "{pedigree_id}", "variant_id": "{ped_3k_id}",  "qty": 5 },

Server computes total, checks credit_used + total ≤ credit_limit, holds the credit, and creates the order in placed status.

8) Office progresses the order

POST /api/orders/{order_id}/accept    // decrements variant stock atomically
POST /api/orders/{order_id}/pack
POST /api/orders/{order_id}/dispatch
POST /api/orders/{order_id}/deliver

Cancel (sales rep, only before accept)

POST /api/orders/{order_id}/cancel
{ "reason": "Store changed their mind" }

End-to-end test recipe

  1. Start Mongo & the backend. Confirm GET /api/health → 200.
  2. Log in as admin (from env-seeded account). Change password.
  3. Create office and sales-rep accounts. Log each in on a separate cookie jar and change passwords.
  4. As office: create categories + subcategories, then products with variants.
  5. As sales_rep: create a store. As office: approve with a credit limit.
  6. As sales_rep: mark attendance (test both success within 100m and failure outside).
  7. As sales_rep: place an order under the credit limit. Try one over — must fail.
  8. As sales_rep: cancel the order. Verify credit_used on the store dropped back to 0.
  9. Place a fresh order. As office: accept → pack → dispatch → deliver. After accept, confirm variant stock decremented.
  10. GET /api/orders/{id} — inspect history[] for the full audit trail.

Products

GET /api/products any user

Query: category_id, subcategory_id, search, page, page_size.

GET /api/products/{product_id} any user

POST /api/products office

Client sends subcategory_id (server denormalises the parent category) and an option_sets block. The server expands the Cartesian product across every non-empty axis and stores one variants[] entry per combination, each seeded at price = base_price, stock = 0, sku = null. Real prices, SKUs and stock are filled via the per-variant PATCH + adjust-stock endpoints after creation.

Body

{
  "name": "Kong Classic Rubber Toy",
  "subcategory_id": "{sub_id}",
  "description": "Chew toy",
  "base_price": 550,
  "discount_price": null,
  "option_sets": {
    "size":   ["Small", "Medium", "Large"],
    "color":  ["Red", "Blue"],
    "weight": null
  }
}

Rules

  • Supported axes: size, weight, color. Other axes require a code change.
  • Empty arrays and duplicate values are dropped silently.
  • All axes empty or option_sets omitted → 1 default variant with all axes null (single-SKU product).
  • PATCH /api/products/{id} does not re-expand — to add a new size after launch use POST /api/products/{id}/variants.

Response 201 (excerpt)

  "_id": "6a...", "name": "Kong...",
  "subcategory_id": "...", "subcategory_name": "Chew",
  "category_id":    "...", "category_name":    "Toys",
  "base_price": 550,
  "variants": [

Inventory

Stock lives in its own collection, one document per variant. Read the section below on the reserve / commit / release lifecycle before wiring the frontend.

Reserve on place, commit on accept, release on cancel.

  • Sales rep places an order → each line's reserved_quantity increments. If any line has available = on_hand - reserved < qty, the whole order is refused (409).
  • Office accepts → each line's quantity_on_hand AND reserved_quantity both decrement (the reservation "commits" into a real consumption).
  • Sales rep or office cancels (only while placed) → each line's reserved_quantity decrements, releasing the hold. on_hand is untouched.
  • Available for the frontend = on_hand - reserved. Server includes this on every variant response.

Credit-limit change workflow

Any change to a store's credit_limit after initial approval requires admin approval. Office proposes → admin approves or rejects. Store carries pending_credit_limit and credit_change_status (none / pending / approved / rejected).

PATCH /api/stores/{store_id}/credit-limit office

Propose a new credit-limit. Parks the value in pending_credit_limit with credit_change_status: "pending". Overwrites any prior pending proposal.

Body

{ "credit_limit": 250000, "reason": "Store hit ₹5L monthly revenue" }

POST /api/stores/{store_id}/credit-limit/approve admin

Applies the pending value as the new credit_limit, clears pending_credit_limit, sets credit_change_status: "approved".

POST /api/stores/{store_id}/credit-limit/reject admin

Clears pending_credit_limit, sets credit_change_status: "rejected". credit_limit unchanged.

Body

{ "reason": "Store's outstanding balance is too high" }

Visits

Field-visit log (replaces the earlier /api/attendance). Multiple visits per store per day are now allowed. Every visit is one of two modes:

  • in_store — GPS mandatory, 100 m geo-fence enforced, optional check_in/check_out for duration tracking.
  • remote — GPS not required, distance check skipped. For phone/WhatsApp orders and other remote interactions.

Every visit has an outcome: order_placed (requires order_id) or no_order (requires no_order_reason). A snapshot of the store's location is stored on every visit for district-wise analytics.

POST /api/visits sales_rep

Body — in-store, no order

{
  "store_id": "{store_id}",
  "mode": "in_store",
  "outcome": "no_order",
  "lat": 12.9716,
  "lng": 77.5946,
  "check_in":  "2026-07-19T10:00:00Z",
  "check_out": "2026-07-19T10:30:00Z",
  "no_order_reason": "Store closed for stock-taking",
  "remarks": "Left samples"
}

Body — remote, with order

{
  "store_id": "{store_id}",
  "mode": "remote",
  "outcome": "order_placed",
  "order_id": "{order_id}",
  "remarks": "WhatsApp order at 8pm"
}

Response 201 (excerpt)

{
  "mode": "in_store",
  "outcome": "no_order",
  "distance_meters": 12.4,
  "duration_minutes": 30,
  "order_value": null,
  "location_snapshot": {
    "location": "MG Road, Bengaluru",
    "district": "Bengaluru South",
    "geo_lat": 12.9716, "geo_lng": 77.5946
  }
}

Errors

  • 404 — store not found or not assigned to caller
  • 400 — store not approved / outside 100 m (in_store only)
  • 422 — payload/mode mismatch (in_store without GPS, order_placed without order_id, etc.)

GET /api/visits any user

Sales rep sees only their own records. Office/admin can filter freely. Query: store_id, user_id, visit_date (YYYY-MM-DD), mode, outcome, district, page, page_size.

GET /api/orders/{order_id} placing sales_rep or office

Returns the full order incl. history[] — every status transition with actor + timestamp. This is what the office order detail UI should render as a step tracker.

POST /api/orders sales_rep

Place an order. Server reserves inventory on every line first, then (if credit fits) holds the credit and persists the order as placed. If the total exceeds available credit, the order is still created as pending_admin_approval — inventory is reserved but credit is NOT held. Admin approves or rejects it.

At creation, the order snapshots the store's district, delivery address (contact + geo + gst), and commercial terms (credit_period_days, cancellation policy, return window). Future store edits don't rewrite existing orders.

Errors

  • 404 — store not assigned to caller
  • 400 — store not approved / variant not found / inactive product or variant / insufficient stock
  • 409 — inventory changed during reservation (retry)

Note: credit exceeded no longer returns 400 — instead the order lands in pending_admin_approval.

POST /api/orders/{order_id}/cancel placing sales_rep or office

Only allowed while status is placed. Releases both the inventory reservations and the credit hold. Records cancel_reason.

Body

{ "reason": "Store changed their mind" }

POST /api/orders/{order_id}/admin-approve admin

Approves a pending_admin_approval order. Re-checks credit (another paid order could have consumed room since placement); if it still doesn't fit, returns 409 and admin can either reject or raise the store's credit_limit. On success: credit is held, status becomes placed.

POST /api/orders/{order_id}/admin-reject admin

Rejects a pending_admin_approval order with a mandatory reason. Releases inventory reservations (credit was never held). Status becomes cancelled with rejection_reason recorded.

Body

{ "reason": "Store's outstanding balance is too high" }

POST /api/orders/{order_id}/accept office

Accept a placed (or delayed) order. Optionally takes a body to reduce quantities on existing lines — the difference is released to inventory and credit. Sets accepted_by_id/accepted_by_name. Cannot add lines or increase qty above qty_ordered.

Body — empty (accept as-is)

{}

Body — reduce a line's qty

{
  "lines": [
    { "product_id": "{pedigree_id}", "variant_id": "{ped_3k_id}", "qty": 3 }
  ],
  "note": "Only 3 available at supplier"
}

Lines omitted from the body default to qty_accepted = qty_ordered. Every accepted line records qty_accepted; the order's total reflects the accepted quantities. Inventory is committed at the accepted qty; the surplus (qty_ordered − qty_accepted) is released.

POST /api/orders/{order_id}/delay office

Mark an active order (placed / accepted / packing / out_for_delivery) as delayed with a mandatory reason. From delayed, the office can resume to any legal next state; delay_reason is cleared once accepted again.

Body

{ "reason": "Awaiting stock top-up from warehouse" }

POST /api/orders/{order_id}/pack office

accepted → packing

POST /api/orders/{order_id}/dispatch office

packing → out_for_delivery

POST /api/orders/{order_id}/deliver office

out_for_delivery → delivered. Stamps delivered_at and computes payment_due_date = delivered_at + credit_period_days from the snapshotted term.

POST /api/orders/{order_id}/payment office/admin

Record a payment. Refuses amount ≤ 0 and overpayment. Derives payment_status from the running total (pending / partially_paid / paid). Releases the paid amount from the store's credit_used (accounts receivable).

Body

{ "amount": 3000, "method": "UPI", "notes": "UPI ref UPI/2026/A1B2" }
